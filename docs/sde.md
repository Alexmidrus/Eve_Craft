# Модуль SDE

## Назначение

Модуль `platform.sde` в Eve Craft теперь отвечает за два связанных набора локальных данных:

- SQLite-базу SDE, импортированную из официального JSONL-архива CCP
- локальный набор изображений `IEC Types`, где для каждого `type_id` доступны PNG-файлы размером `32` и `64` пикселя

Оба ресурса подготавливаются автоматически при первом запуске приложения. На следующих стартах модуль адресно восстанавливает только отсутствующий ресурс: базу SDE, каталог `IEC Types` или оба сразу, если локальных данных нет совсем.

## Что хранится локально

После синхронизации модуль поддерживает:

- `runtime/databases/sde.sqlite3` - рабочую базу SDE
- `runtime/resources/sde/types/` - локальный каталог `Types`-изображений
- `runtime/downloads/` - временные ZIP-архивы во время загрузки
- `runtime/tmp/` - временные SQLite-базы и временные каталоги распаковки

Каталог изображений содержит файлы вида:

- `34_32.png`
- `34_64.png`
- `587_32.png`
- `587_64.png`

Рядом с изображениями хранится `manifest.json`, по которому модуль понимает, какая версия набора уже установлена локально и сколько PNG-файлов должно лежать в каталоге. Если manifest отсутствует или фактическое число файлов не совпадает с `image_count`, набор считается поврежденным и загружается заново.

## Разделение по слоям

- `domain` содержит модели состояния: версии SDE, версии набора изображений, общий статус и результаты синхронизации
- `application` содержит orchestration-логику: отдельно для базы SDE, отдельно для IEC Types, и общий coordinator для обоих ресурсов
- `infrastructure` содержит HTTP-клиенты, импортёры архивов, работу с SQLite и manifest-файлами
- `presentation` содержит существующий Qt-диалог проверки и обновления SDE
- `service.py` выступает фасадом для остального приложения

## Основные компоненты

### `SdeService`

`SdeService` - публичная точка входа для контейнера приложения. Он собирает:

- `SdeSynchronizer` для базы SDE
- `TypeImageCollectionSynchronizer` для `IEC Types`
- `SdeResourceSynchronizer` как общий coordinator

Сервис по-прежнему предоставляет основной API:

- `get_status(refresh_remote=False)`
- `ensure_ready(report_progress)`
- `update(report_progress, force=False)`
- `describe_status()`

Дополнительно сервис умеет отдавать путь к локальному изображению:

- `type_image_path(type_id, size=64)`

### `SdeSynchronizer`

`SdeSynchronizer` продолжает отвечать только за SQLite-базу SDE:

1. Читает локально установленную версию из `sde_catalog_info`.
2. Определяет актуальный удаленный билд через `latest.jsonl`.
3. При необходимости скачивает ZIP-архив SDE.
4. Импортирует JSONL-файлы во временную SQLite-базу.
5. Атомарно активирует новую базу через `os.replace`.

### `TypeImageCollectionSynchronizer`

`TypeImageCollectionSynchronizer` отвечает только за локальный набор `IEC Types`:

1. Читает локальный `manifest.json`.
2. Проверяет, есть ли в каталоге хотя бы один импортированный PNG-файл.
3. Получает актуальный URL архива `Types` из опубликованного `iec/index.md`.
4. Сравнивает удаленный архив с локально установленным набором по URL и доступным заголовкам `ETag`, `Last-Modified`, `Content-Length`.
5. Если нужен апдейт, скачивает ZIP-архив.
6. Распаковывает `Types/*.png` во временный каталог.
7. Активирует новый каталог ресурсов и записывает новый manifest.

### `SdeResourceSynchronizer`

`SdeResourceSynchronizer` объединяет оба потока в единый lifecycle для приложения:

- `get_status()` возвращает объединенный статус базы и изображений
- `ensure_ready()` проверяет локальное наличие SDE и `IEC Types` и подгружает только отсутствующий ресурс
- `update()` обновляет оба ресурса через один progress stream

Именно этот coordinator делает поведение модуля единым: снаружи приложение видит один сервис `container.sde`, хотя внутри синхронизируются два разных набора данных.

## Источники данных

### SDE база

Для базы SDE модуль использует официальный JSONL endpoint CCP:

- `https://developers.eveonline.com/static-data/tranquility/latest.jsonl`

На его основе строится URL архива вида:

- `https://developers.eveonline.com/static-data/tranquility/eve-online-static-data-<build>-jsonl.zip`

### IEC Types

Для изображений используется опубликованный индекс IEC из репозитория `esi-docs`:

- `https://raw.githubusercontent.com/esi/esi-docs/main/docs/services/iec/index.md`

Из этого Markdown-индекса извлекается актуальная ссылка на архив:

- `https://web.ccpgamescdn.com/aws/developers/Uprising_V21.03_Types.zip`

Важно: IEC помечен как deprecated, но этот индекс по-прежнему публикует актуальные ZIP-сборки для полного локального импорта изображений.

## Сценарий импорта базы SDE

Импорт базы выполняется в `SdeImporter` во временный файл:

- `runtime/tmp/sde_build_<build>.sqlite3`

Во время импорта создаются таблицы SQLAlchemy и последовательно загружаются ключевые JSONL-наборы:

- категории, группы, market groups, meta groups
- типы предметов и локализованные названия
- dogma units, attributes, effects и связи type dogma
- материалы переработки, compressible types
- blueprints и данные по activity/materials/products/skills
- карта мира: регионы, созвездия, солнечные системы
- NPC corporations, station operations, NPC stations
- planetary schematics

После успешного импорта в таблицу `sde_catalog_info` записываются:

- номер билда
- дата релиза
- время импорта
- URL архива
- `ETag`
- `Last-Modified`

## Сценарий импорта IEC Types

Импорт изображений выполняется в `TypeImageCollectionImporter` во временный каталог внутри `runtime/tmp/`.

Что происходит:

1. Открывается ZIP-архив `Types`.
2. Из него выбираются только файлы `Types/*.png`.
3. Каждый файл валидируется по имени: ожидается шаблон `<type_id>_<size>.png`, где `size` равен `32` или `64`.
4. Файлы распаковываются в плоский локальный каталог.
5. После успешной распаковки каталог активируется как рабочий ресурсный каталог.
6. Репозиторий записывает `manifest.json` с метаданными установленного набора.

Таким образом, приложению не нужно делать web-запрос за каждым изображением отдельно: после синхронизации все нужные PNG уже лежат на диске локально.

## Поведение при старте приложения

На старте `ApplicationStartupService` вызывает:

- `container.sde.ensure_ready(...)`

Дальше модуль ведет себя адресно:

- если локальная база и изображения уже есть, приложение сразу продолжает старт
- если отсутствует только база, загружается и устанавливается только SDE
- если отсутствуют только изображения, загружается и устанавливается только каталог `IEC Types`
- если отсутствуют оба ресурса, модуль загружает оба
- если удаленная проверка недоступна, но локальная версия уже есть, приложение продолжает работу с локальными данными и пишет warning
- если удаленная проверка недоступна и локальной версии нет, старт завершается ошибкой

Порядок старта такой:

1. Инициализация application database
2. Проверка наличия SDE SQLite
3. Проверка наличия `IEC Types`
4. Загрузка только недостающих ресурсов

## Обновление из интерфейса

Новый UI для изображений не добавлялся.

Существующий `SdeUpdateDialogController` продолжает работать через тот же `SdeService`, поэтому ручное обновление теперь фактически обновляет сразу оба ресурса:

- базу SDE
- локальный набор `IEC Types`

Диалог показывает общий прогресс и лог, а детали по изображениям приходят в тот же поток сообщений `OperationProgress`.

## Устойчивость к ошибкам

В модуле предусмотрены такие защитные сценарии:

- если сеть недоступна, но локальная база SDE уже есть, приложение использует ее дальше
- если сеть недоступна, но локальный набор изображений уже есть и число PNG совпадает с manifest, приложение использует локальные PNG дальше
- если апдейт одного из ресурсов падает, но предыдущая локальная версия доступна, приложение может продолжить работу с ней
- временные ZIP-архивы удаляются после завершения операции
- временные SQLite-базы и временные каталоги распаковки очищаются при ошибках

## Как расширять дальше

Если позже понадобятся другие IEC-наборы, например `Icons` или `Renders`, лучше повторять ту же схему:

1. Добавить отдельный клиент, который умеет найти нужный архив у CCP.
2. Добавить отдельный importer и repository для локального каталога.
3. Подключить новый synchronizer в общий `SdeResourceSynchronizer`.
4. Не смешивать бизнес-логику использования изображений с инфраструктурой их загрузки.

Если поверх SDE или `IEC Types` появится прикладная логика для конкретного bounded context, ее лучше размещать не в `platform.sde`, а в соответствующем модуле `industry` или `market`.
