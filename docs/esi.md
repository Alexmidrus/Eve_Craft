# Модуль ESI

## Назначение

`platform.esi` в Eve Craft должен стать платформенным интеграционным модулем для работы с EVE Swagger Interface.

Это не бизнес-модуль и не просто HTTP-клиент. Его задача:

- централизованно выполнять запросы к ESI
- работать с публичными и авторизованными endpoint'ами
- учитывать совместимость API, кэширование, лимиты и пагинацию
- подготавливать данные к дальнейшему сохранению
- хранить собственное техническое состояние синхронизации
- отдавать остальному приложению стабильный API для получения игровых данных

Модуль нужен всему приложению, поэтому его место - `src/eve_craft/platform/esi/`.

## Внешние требования ESI, которые влияют на архитектуру

При проектировании модуля нужно опираться не только на внутреннюю архитектуру проекта, но и на правила самой ESI.

Ключевые требования из официальной документации:

- запросы должны использовать `X-Compatibility-Date`, чтобы приложение явно фиксировало ожидаемое поведение API
- ESI ожидает корректную работу с HTTP cache headers, включая `ETag`, `Expires`, `If-None-Match` и сценарий `304 Not Modified`
- лимиты состоят не только из общего ограничения по route group, но и из отдельного error limit бюджета; реакция на `420 Error limited` должна быть встроенной
- у разных endpoint'ов разные схемы пагинации: `X-Pages`, cursor-based и `from_id`
- для cursor-based пагинации возможны дубликаты между страницами, значит дедупликация и checkpointing должны быть частью sync-логики
- клиент должен отправлять корректный `User-Agent` и не опрашивать API агрессивнее, чем требует задача

Официальные страницы:

- [ESI Overview](https://developers.eveonline.com/docs/services/esi/overview/)
- [ESI Rate Limiting](https://developers.eveonline.com/docs/services/esi/rate-limiting/)
- [Cursor-Based Pagination](https://developers.eveonline.com/docs/services/esi/pagination/cursor-based/)
- [X-Pages Pagination](https://developers.eveonline.com/docs/services/esi/pagination/x-pages/)
- [From-ID Pagination](https://developers.eveonline.com/docs/services/esi/pagination/from-id/)
- [ESI Best Practices](https://developers.eveonline.com/docs/services/esi/best-practices/)

Из этого следует важный вывод: `platform.esi` должен быть не "оберткой над requests", а модулем интеграционной оркестрации.

## Место в архитектуре проекта

В контексте текущего modular monolith границы должны быть такими:

- `platform.esi` отвечает за техническую интеграцию с ESI
- `platform.auth` отвечает за OAuth, токены, refresh и безопасное хранение учетных данных
- `platform.db` отвечает за engine, sessions, unit of work и общую SQLite-инфраструктуру
- `modules.industry` и `modules.market` отвечают за свои бизнес-модели и свои таблицы
- `platform.characters` может стать первым платформенным потребителем ESI, если в нем появится управление персонажами, профилями и связью с токенами

Зависимости должны выглядеть так:

- `modules.* -> platform.esi`
- `platform.characters -> platform.esi`
- `platform.esi -> platform.auth`
- `platform.esi -> platform.db`

Запрещенная связность:

- `platform.esi` не должен содержать бизнес-логику industry или market
- `platform.db` не должен становиться универсальным модулем "запиши любые данные куда угодно"
- `modules.industry` и `modules.market` не должны знать про внутренние HTTP-детали ESI

## Что именно должен делать модуль

Полноценный `platform.esi` должен предоставлять следующие возможности:

- выполнение одиночных запросов к публичным и авторизованным endpoint'ам
- централизованную установку служебных заголовков: `User-Agent`, `X-Compatibility-Date`, conditional cache headers
- обработку `401`, `403`, `404`, `420`, `5xx`, сетевых ошибок и временных отказов
- контроль rate limit и error limit без копирования логики по модулям
- унифицированную пагинацию для `single`, `x_pages`, `cursor`, `from_id`
- хранение checkpoint'ов синхронизации
- возврат нормализованного результата: payload плюс метаданные ответа
- подготовку данных для сохранения: разбиение на страницы, нормализацию envelope, передачу в consumer/sink
- собственный технический статус: здоровье клиента, остатки лимитов, время последней успешной синхронизации, активные checkpoint'ы

Что модуль делать не должен:

- рассчитывать прибыль, маржу, производственные цепочки или другие бизнес-правила
- хранить все бизнес-таблицы приложения у себя
- смешивать данные нескольких bounded context в одну универсальную схему
- выполнять обогащение данными SDE внутри транспортного слоя

## Варианты архитектуры записи в БД

### Вариант 1. `platform.esi` сам пишет полученные бизнес-данные в БД

Идея:

- `platform.esi` делает HTTP-запросы
- сам разбирает ответы
- сам записывает результаты в SQLite
- верхние модули получают уже сохраненные данные

Плюсы:

- быстрее стартовать с первыми интеграциями
- вся логика "получил и сохранил" находится в одном месте
- удобно для небольшого прототипа

Минусы:

- `platform.esi` быстро превратится в крупный центральный модуль со знанием про персонажей, активы, ордера, jobs, wallet, industry и market
- нарушится правило владения данными: платформенный технический модуль начнет владеть бизнес-схемой
- при росте проекта появится сильная связность между `esi` и бизнес-модулями
- изменения в одной предметной области будут приводить к изменениям в сервисном модуле
- усложнится тестирование и миграции схемы

Вывод:

Этот вариант допустим только для очень маленького проекта или короткого MVP. Для Eve Craft с прицелом на рост он слишком рискованный.

### Вариант 2. `platform.esi` только получает и подготавливает данные, а запись выполняется вне него

Идея:

- `platform.esi` отвечает за transport, auth-aware доступ, лимиты, кэширование, пагинацию и sync checkpoint'ы
- он возвращает нормализованные данные и технические метаданные
- отдельный слой выполняет mapping в локальные таблицы и запись в БД

Плюсы:

- сохраняются чистые границы ответственности
- каждый модуль владеет своими таблицами и своими репозиториями
- `platform.esi` можно расширять новыми endpoint'ами без разрастания бизнес-логики
- упрощаются тесты: ESI можно мокать как внешний источник, а модули тестировать как отдельных потребителей
- легче добавлять новые bounded context без изменения уже существующих схем хранения

Минусы:

- потребуется чуть больше инфраструктурного кода
- нужно заранее продумать формат обмена между `platform.esi` и потребителями
- если спроектировать слабый публичный API, можно получить дублирование mapping-кода в нескольких местах

Вывод:

Этот вариант лучше соответствует архитектуре проекта, но в "чистом" виде ему не хватает одного нюанса: самому `platform.esi` все равно нужно хранить свое техническое состояние.

### Рекомендуемый вариант. Гибридная схема

Рекомендуется использовать вариант 2, но с важным уточнением:

- `platform.esi` не должен писать бизнес-данные всех модулей
- `platform.esi` должен иметь право хранить свои собственные технические данные

То есть:

- данные бизнес-смыслового уровня пишут владельцы этих данных
- данные интеграционного уровня пишет сам `platform.esi`

Это лучший баланс между ростом проекта и практичностью.

## Что именно может хранить сам `platform.esi`

Только технические и интеграционные сущности:

- metadata по route и endpoint'ам
- cache metadata: `ETag`, `Expires`, `Last-Modified`, hash тела, момент последнего запроса
- checkpoint'ы синхронизации по route, character, corporation или другому контексту
- watermarks для `from_id`
- cursor state для cursor-based endpoint'ов
- журнал sync-запусков и их статусы
- ошибки последней синхронизации
- при необходимости - сырые snapshot'ы ответов как опциональный staging/landing слой

Что не должен хранить `platform.esi`:

- рыночные модели модуля `modules.market`
- производственные или проектные модели модуля `modules.industry`
- агрегаты, рассчитанные с участием SDE и бизнес-правил

## Наиболее подходящая модель для Eve Craft

Для текущего проекта оптимальна такая схема:

1. `platform.esi` строится как общий ingestion-модуль.
2. `platform.auth` поставляет токены и выполняет refresh.
3. `platform.db` дает только инфраструктуру транзакций и SQLAlchemy session/uow.
4. `platform.esi` хранит только техническое состояние интеграции.
5. Конкретные игровые данные сохраняются в модуле-владельце.

Примеры владения:

- данные о привязанных персонажах и их базовом профиле - `platform.characters`
- данные market orders, истории рынка, торговых снимков - `modules.market`
- данные industry jobs, assets, blueprints, structures, projects - `modules.industry`

Именно этот вариант лучше всего соответствует правилам из `AGENTS.md` и масштабируется без превращения `platform.esi` в dump-модуль.

## Целевая структура `platform.esi`

Предлагаемая структура:

```text
src/eve_craft/platform/esi/
  application/
    contracts.py
    request_executor.py
    sync_orchestrator.py
    page_collectors.py
    checkpointing.py
  domain/
    models.py
    pagination.py
    rate_limits.py
    cache_policy.py
    scopes.py
    errors.py
  infrastructure/
    http_client.py
    auth_provider.py
    cache_repository.py
    sync_state_repository.py
    models.py
    route_catalog.py
    throttling.py
  service.py
```

`presentation/` сейчас не обязателен. Его стоит добавлять только тогда, когда появится UI для статуса ESI, логина, диагностики лимитов или ручных sync-операций.

## Роли слоев

### `domain`

Содержит чистые модели и правила интеграции:

- описание route и требований к scope
- тип пагинации
- cache policy
- модели rate limit состояния
- checkpoint'ы и sync state
- ошибки уровня домена интеграции

### `application`

Содержит orchestration:

- выполнение запроса через набор портов
- выбор стратегии пагинации
- обновление checkpoint'ов
- потоковую выдачу страниц или item batches
- координацию fetch-and-hand-off сценариев

### `infrastructure`

Содержит адаптеры:

- HTTP-клиент для ESI
- адаптер к `platform.auth`
- SQLAlchemy-репозитории технического состояния ESI
- persistence-модели таблиц ESI metadata
- throttling/backoff реализацию
- каталог поддерживаемых route/operation descriptor'ов

### `service.py`

Остается узким фасадом, который собирает модуль и отдает публичный API приложению.

## Ключевые абстракции

Чтобы модуль не разрастался бесконтрольно, стоит сразу ввести несколько опорных абстракций.

### `EsiOperationDescriptor`

Описывает поддерживаемую операцию:

- HTTP method и path template
- нужен ли access token
- какие scope обязательны
- какой тип пагинации используется
- какая cache policy применяется
- как строится context key для checkpoint'ов

Это позволит не размазывать знание о поведении endpoint'ов по десяткам файлов.

### `EsiRequestContext`

Контекст выполнения запроса:

- character/account context
- route params
- query params
- correlation id
- режим обновления: использовать cache, принудительно обновить, догнать синхронизацию

### `EsiResponseEnvelope`

Нормализованный результат запроса:

- `payload`
- `requested_at`
- `status_code`
- `etag`
- `expires_at`
- `rate_limit_snapshot`
- `pagination_state`
- `source`: `network` или `cache`

Именно в таком виде данные удобно отдавать дальше на mapping и сохранение.

### `PaginationStrategy`

Отдельные реализации для:

- single page
- `X-Pages`
- cursor-based
- `from_id`

Это одна из ключевых точек расширения модуля.

### `PayloadSink`

Порт для получателя данных.

Примеры:

- sink, который пишет технический snapshot
- sink, который передает страницы в `modules.market`
- sink, который записывает character assets в модуль-владелец

Такой подход разделяет "получение" и "сохранение", не теряя удобства.

## Рекомендуемый поток данных

Целевой сценарий должен выглядеть так:

1. Бизнес-модуль или платформенный потребитель инициирует sync-операцию.
2. `platform.esi` выбирает descriptor операции.
3. Через порт к `platform.auth` модуль получает валидный token, если endpoint авторизованный.
4. HTTP-клиент выполняет запрос с `X-Compatibility-Date`, `User-Agent` и cache headers.
5. `platform.esi` обрабатывает rate limit, cache metadata и пагинацию.
6. Для каждой страницы формируется `EsiResponseEnvelope`.
7. `platform.esi` обновляет свои checkpoint'ы и technical metadata.
8. Дальше данные передаются в consumer/sink.
9. Consumer выполняет mapping в собственные таблицы и коммитит свою транзакцию.

Для SQLite и больших наборов данных желательно сохранять результат page-by-page или chunk-by-chunk, а не одной гигантской транзакцией.

## Особенности пагинации, которые нужно заложить сразу

### `X-Pages`

Для таких endpoint'ов модуль должен:

- сделать первый запрос
- прочитать `X-Pages`
- сформировать план обхода страниц
- сохранять прогресс по номеру последней успешно обработанной страницы

### Cursor-based

Для таких endpoint'ов модуль должен:

- хранить `before` и `after` курсоры
- уметь продолжать прерванный обход
- быть готовым к дубликатам между страницами
- хранить dedup/watermark стратегию в sync state или на стороне consumer'а

### `from_id`

Для таких endpoint'ов модуль должен:

- хранить последний обработанный watermark
- понимать направление обхода
- поддерживать incremental sync без полного reread набора

## Взаимодействие с `platform.auth`

Важно не смешивать ответственность двух платформенных модулей:

- `platform.auth` владеет OAuth flow, refresh token, access token и безопасным хранением
- `platform.esi` не должен знать детали UI-логина и хранения секретов
- `platform.esi` работает через порт наподобие `TokenProvider`

Это позволит развивать авторизацию отдельно от сетевой логики ESI.

## Взаимодействие с `platform.db`

`platform.db` должен оставаться инфраструктурным основанием:

- engine
- session factory
- unit of work
- общие таблицы приложения, если они действительно shared

Но `platform.db` не должен становиться центральным бизнес-слоем записи ESI-данных.

Правильная модель:

- `platform.db` предоставляет SQLAlchemy infrastructure
- `platform.esi.infrastructure` содержит свои технические репозитории
- владельцы данных содержат свои бизнес-репозитории у себя

## Что можно сделать опционально позже

Если со временем появится потребность в replay, диагностике и пересборке агрегатов без повторного вызова ESI, можно добавить опциональный landing слой:

- таблицу или набор таблиц с raw JSON snapshot'ами
- привязку snapshot к route, character, времени и cache metadata
- TTL или политику очистки старых payload

Но включать хранение всех raw payload по умолчанию на старте не обязательно. Для desktop-приложения на SQLite это лучше делать адресно.

## Этапы внедрения

### Этап 1. Базовый каркас

- конфиг `base_url`, `compatibility_date`, `user_agent`, timeout
- HTTP client
- error handling
- интеграция с `platform.auth`
- фасад `EsiService`

### Этап 2. Интеграционная устойчивость

- cache metadata
- rate limit tracking
- retry/backoff
- operational status

### Этап 3. Пагинация и checkpointing

- `X-Pages`
- cursor-based
- `from_id`
- таблицы sync state

### Этап 4. Первый потребитель

Лучше всего подключать не сразу тяжелый business workflow, а первый относительно простой consumer:

- `platform.characters` для профиля персонажа и базовых account-bound данных

После этого можно подключать `modules.market` и `modules.industry`.

## Итоговая рекомендация

Для Eve Craft наиболее подходящее решение:

- строить `platform.esi` как платформенный модуль интеграции и синхронизации
- не отдавать ему владение бизнес-таблицами приложения
- разрешить ему хранить только собственное техническое состояние
- сохранять игровые данные в том bounded context, которому они принадлежат
- использовать `platform.db` только как инфраструктуру БД, а не как универсальный сервис записи

Итоговый принцип можно сформулировать так:

`platform.esi` отвечает за то, чтобы данные из ESI корректно, безопасно и масштабируемо пришли в приложение.

Но за смысл этих данных и их долгосрочное хранение должен отвечать модуль-владелец.
