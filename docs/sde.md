# Модуль SDE

## Назначение

Модуль `platform.sde` отвечает за подготовку и поддержку локальной базы данных SDE для Eve Craft. Он проверяет установленную версию, сравнивает ее с актуальной сборкой на стороне CCP, скачивает архив, импортирует нужные JSONL-справочники в SQLite и активирует новую базу для дальнейшей работы приложения.

Итог работы модуля - локальный файл `runtime/databases/sde.sqlite3`, который можно использовать как источник справочных данных для других частей системы.

## Где находится

```text
src/eve_craft/platform/sde/
  application/
  domain/
  infrastructure/
  presentation/
  service.py
```

Модуль организован по слоям:

- `domain` содержит модели состояния SDE: удаленная версия, установленная версия, статус и результат синхронизации
- `application` содержит сценарий синхронизации и правила переходов между состояниями
- `infrastructure` содержит работу с сетью, архивом, импортом JSONL и локальной SQLite-базой
- `presentation` содержит Qt-диалог проверки и обновления SDE
- `service.py` собирает зависимости и дает внешний API для приложения

## Основные компоненты

### `SdeService`

`SdeService` - входная точка для остального приложения. Он создает:

- `SdeMetadataRepository` для чтения и активации локальной базы
- `EveStaticDataClient` для получения метаданных и скачивания архива
- `SdeImporter` для импорта JSONL в SQLite
- `SdeSynchronizer` как главный слой оркестрации

Сервис предоставляет три основных метода:

- `get_status(refresh_remote=False)` - вернуть локальный статус и при необходимости проверить удаленную версию
- `ensure_ready(report_progress)` - убедиться, что локальная база готова к использованию во время старта приложения
- `update(report_progress, force=False)` - принудительно пройти сценарий обновления

### `SdeSynchronizer`

`SdeSynchronizer` управляет жизненным циклом SDE:

1. Читает локально установленную версию из `sde_catalog_info`.
2. При необходимости запрашивает последнюю доступную сборку через `latest.jsonl`.
3. Сравнивает номера билдов локальной и удаленной версии.
4. Если обновление не нужно, возвращает статус "up to date".
5. Если обновление нужно, скачивает ZIP-архив SDE.
6. Проверяет, что билд внутри архива совпадает с ожидаемым.
7. Импортирует данные во временную SQLite-базу.
8. Атомарно заменяет рабочую базу через `os.replace`.
9. Удаляет временные файлы и архив после завершения.

Синхронизатор также преобразует внутренний прогресс скачивания и импорта в общий `OperationProgress`, чтобы UI и стартовый экран могли показывать ход операции.

## Источник данных и проверка версии

Для определения актуальной версии модуль использует официальный JSONL-метафайл:

- `https://developers.eveonline.com/static-data/tranquility/latest.jsonl`

После этого строится URL архива вида:

- `https://developers.eveonline.com/static-data/tranquility/eve-online-static-data-<build>-jsonl.zip`

Дополнительно клиент делает `HEAD`-запрос к архиву, чтобы сохранить `ETag` и `Last-Modified` в метаданных локальной базы.

## Сценарий импорта

Импорт выполняется в `SdeImporter` во временную базу `runtime/tmp/sde_build_<build>.sqlite3`. Перед началом создаются таблицы SQLAlchemy-моделей, затем JSONL-файлы из архива обрабатываются в фиксированной последовательности.

Сейчас импортируются данные из следующих наборов:

- категории, группы, market groups и meta groups
- типы предметов и связанные локализованные названия/описания
- dogma units, attributes, effects и связи type dogma
- материалы переработки и randomized reprocessing materials
- compressible types
- blueprints, activities, materials, products и skills
- регионы, созвездия и солнечные системы
- NPC corporations, station operations и NPC stations
- schematics для planetary industry

После импорта в таблицу `sde_catalog_info` записываются:

- номер билда
- дата релиза
- время импорта
- URL архива
- `ETag`
- `Last-Modified`

Если архив неполный и в нем отсутствуют обязательные JSONL-файлы, импорт завершается ошибкой.

## Где хранятся файлы

Во время работы модуль использует несколько путей из `AppConfig`:

- `runtime/databases/sde.sqlite3` - активная рабочая база SDE
- `runtime/downloads/` - временно скачанный ZIP-архив
- `runtime/tmp/` - временная база во время импорта

Файл архива удаляется после завершения операции. Временная база тоже удаляется, если активация не удалась.

## Поведение при старте приложения

На старте `ApplicationStartupService` вызывает `container.sde.ensure_ready(...)`.

Поведение такое:

- если локальная база уже есть и билд актуален, приложение использует ее без повторного импорта
- если локальная база устарела, модуль автоматически скачивает и устанавливает новый билд
- если удаленную версию нельзя проверить, но локальная база уже существует, приложение продолжает работу с локальной версией и пишет предупреждение
- если удаленная проверка недоступна и локальной базы нет, старт завершается ошибкой, потому что приложению нечего использовать как SDE

## Обновление из интерфейса

Для ручной работы существует `SdeUpdateDialogController` в слое `presentation`.

Диалог умеет:

- показать установленный билд, последний известный билд и дату релиза
- отдельно запустить проверку статуса
- запустить обновление SDE
- показывать лог и прогресс в фоне без блокировки UI

Долгие операции выполняются в `QThread` через `BackgroundTaskWorker`, поэтому окно остается отзывчивым.

## Устойчивость к ошибкам

В текущей реализации модуль уже покрывает несколько важных сценариев отказа:

- при сетевой ошибке во время проверки версии используется установленная локальная база, если она есть
- при неудачном обновлении приложение может продолжить работу со старой локальной базой
- при поврежденном или неподходящем архиве новая база не активируется
- временные артефакты очищаются в блоке `finally`

Это поведение дополнительно закреплено тестами в `tests/platform/sde`.

## Как расширять модуль

Если нужно добавить новые справочники в импорт, рабочий путь такой:

1. Добавить SQLAlchemy-модели в `infrastructure/models.py`.
2. Добавить новый JSONL-файл в `SdeImporter.IMPORT_SEQUENCE`.
3. Реализовать соответствующий метод `import_*`.
4. При необходимости расширить доменные или прикладные сервисы, которые будут использовать новые таблицы.
5. Добавить тесты на импорт и синхронизацию в `tests/platform/sde`.

Новые бизнес-правила поверх SDE лучше размещать не в `platform.sde`, а в том модуле, который владеет конкретным сценарием использования данных.
